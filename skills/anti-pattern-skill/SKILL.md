# 反模式审查

## 概览

开始前先阅读 [references/anti-patterns-checklist.md](references/anti-patterns-checklist.md)。

目标是识别那些局部看似在解决问题、长期却会损害可变更性、可靠性或交付效率的做法，并输出结构化审查报告与整改待办。

## 审查流程

1. 先理解仓库结构、核心执行路径、职责边界和共享抽象
2. 对照清单检查编程、方法论、测试层面的反模式
3. 只报告证据充分、足以支持纠正行动的发现
4. 如果范围较大，允许并行子 Agent 分工审查

## 判断标准

每条发现都必须：
- 引用支持它的文件路径、测试、文档或项目产物
- 说明什么局部行为或结构让它看起来像一个“解决方案”
- 说明它随着时间推移会带来的更大成本
- 区分观察到的事实与推断
- 给出能够渐进式降低伤害的纠正方向

避免只基于代码风格偏好、框架口味或孤立单行代码形成薄弱结论。

当反模式已经实质性损害可变更性、正确性风险、交付信心或架构清晰度时，使用 `高`。对于明确存在、但影响范围有限的可维护性或测试债务，使用 `中`。对于真实存在但次要的问题，使用 `低`。

置信度应反映证据量：
- `高`：多个位置的直接证据，或很强的结构性证据
- `中`：局部证据清晰，但仓库覆盖范围有限
- `低`：担忧看起来合理，但证据仍不完整

优先关注那些最明显损害可变更性、可靠性或团队交付速度的反模式。

## 输出格式

始终产出两个部分，先给发现，再给整改待办。

### 1. 审查发现

按严重级别排序，再按置信度和可能影响排序列出发现。每条发现都要简洁、基于证据。

使用如下结构：

```md
## 审查发现

### 1. 简短的发现标题
- 反模式：过早优化
- 类别：方法论
- 严重级别：高
- 置信度：中
- 证据：`src/cache.ts` 和 `docs/design.md` 引入了多层缓存，但没有已测量的性能瓶颈，也没有回退失效策略。
- 影响：团队现在要维护的失效路径比已被证明的延迟收益更多，复杂度和缺陷风险都被抬高了。
- 建议的纠正方向：删除推测性的缓存层，保留一条有测量依据的优化路径，并在继续调优前补上基准测试。
```

### 2. 按严重级别排序的整改待办

把审查发现翻译成一个按执行顺序排列的行动清单。

每个待办项都应包含：
- 优先级
- 目标区域
- 目标
- 建议的第一步
- 预期收益
- 风险说明

优先从低风险、高杠杆的动作开始，比如删除死代码、抽取可复用行为、把配置移出代码、简化过度设计的抽象，或把测试重新平衡到面向可观察行为的层面。

## 不要做什么

- 把“反模式”一词滥用到普通代码清理上
- 在没有点明支撑证据的情况下报告发现
- 在可以渐进修正时直接建议重写
- 仅凭一个孤立片段就推断整个仓库都存在同类问题
- 把有意识的权衡误判为反模式，除非更广泛的伤害已经可见

如果你没有发现高置信度的反模式，就直接说明这一点，并补充你覆盖了哪些审查范围。
